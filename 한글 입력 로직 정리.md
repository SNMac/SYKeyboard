## 전체 아키텍처

```
사용자 터치
    ↓
BaseKeyboardViewController (버튼 이벤트 라우팅)
    ↓
HangeulKeyboardCoreViewController (버퍼 관리 + textDocumentProxy 조작)
    ↓
HangeulProcessable (프로세서 프로토콜)
    ├── inputWithRestore종성() / deleteWithRestore종성()  ← 프로토콜 기본 구현 (종성 복원)
    └── input() / delete()                               ← 각 프로세서 고유 로직
    ↓
HangeulAutomata (표준 한글 오토마타: 자소 합치기/분해)
```

핵심 원칙:
- **프로세서는 버퍼 문자열만 다루고, 화면(textDocumentProxy) 조작은 컨트롤러가 전담**한다.
- **종성 복원은 프로토콜 기본 구현에서 공통 처리**하며, 각 프로세서는 고유 로직만 구현한다.
- **VC는 `input()`이 아닌 `inputWithRestore종성()`을, `delete()`이 아닌 `deleteWithRestore종성()`을 호출**한다.

## 1. 두 개의 버퍼 구조 (O(1) 설계)

```
committedBuffer: "안녕하"    ← 확정된 텍스트, proxy에서 건드리지 않음
composingBuffer: "세"        ← 현재 조합 중, proxy에서 delete→reinsert 대상
                              (항상 최대 1~2글자)
```

화면에 보이는 텍스트: `"안녕하세"`

입력할 때마다 composingBuffer(1~2글자)만 지우고 다시 쓰므로, 전체 텍스트 길이와 무관하게 항상 **O(1)** 성능을 보장한다.

## 2. 공통 타입 정의

### CompositionResult

프로세서가 입력 결과를 컨트롤러에 돌려주는 구조체.

```swift
struct CompositionResult {
    let committed: String              // 확정된 부분 → committedBuffer에 추가
    let composing: String              // 조합 중인 부분 → composingBuffer를 교체
    let input글자: String?             // 반복 입력(길게 누르기)용 마지막 글자
    let consumedCommittedCount: Int    // 프로세서가 committedTail에서 소비한 글자 수 (0, 1, 또는 2)
}
```

`consumedCommittedCount`는 종성 복원 시 사용된다. 프로세서가 `committedTail`의 글자를 composing으로 끌어올렸을 때, VC에 그만큼 `committedBuffer`에서 제거하라고 알린다.

예시:

|상태|입력|committed|composing|consumedCommittedCount|설명|
|---|---|---|---|---|---|
|`"간"`|`"ㅏ"`|`"가"`|`"나"`|0|연음|
|`"가"`|`"ㄴ"`|`""`|`"간"`|1|종성 복원 (committed "가" 소비)|
|`"가"`|`"1"`|`"가1"`|`""`|0|비한글 확정|

편의 생성자:
- `CompositionResult.composingOnly("한")` → committed 없이 composing만
- `CompositionResult.commitAll("가1")` → 조합 없이 전체 확정

### DeleteResult

삭제 결과를 반환하는 구조체. composing 변경뿐만 아니라 committed 소비까지 포함한다.

```swift
struct DeleteResult {
    let composing: String              // 삭제 후 남은 조합 문자열
    let consumedCommittedCount: Int    // 프로세서가 committedTail에서 소비한 글자 수 (0, 1, 또는 2)
}
```

예시:

|상태 (composing / committedTail)|결과 composing|consumedCommittedCount|설명|
|---|---|---|---|
|`"ㄴ"` / `"가"`|`"간"`|1|종성 복원|
|`"ㄷ"` / `"간ㆍ"`|`"간ㆍㄷ"`|2|천지인 비표준 모음 연음|
|`"간"` / `"가"`|`"가"`|0|일반 삭제|

### SpaceInputResult

```swift
enum SpaceInputResult {
    case insertSpace        // 실제 공백 " " 입력 (두벌식/나랏글)
    case commitCombination  // 조합만 끊고 공백 없음 (천지인)
}
```

## 3. 프로토콜 기본 구현 — 종성 복원

`HangeulProcessable` 프로토콜의 extension에서 종성 복원 로직을 공통으로 처리한다. 각 프로세서는 `input()`과 `delete()`만 구현하면 되고, VC는 이 기본 구현인 `inputWithRestore종성()`과 `deleteWithRestore종성()`을 호출한다.

### 3-1. inputWithRestore종성

각 프로세서의 `input()`을 호출한 뒤, 결과에 대해 종성 복원을 시도한다.

```
종성 복원 조건 (모두 만족해야 함):
1. 이전 composing이 있었음 (변환 결과로 자음이 남은 것)
2. result.committed가 비어있음 (프로세서가 확정한 게 없음)
3. result.composing이 낱자 자음 1개
4. committedTail이 비어있지 않음
5. isProtected가 false (스페이스 확정으로 보호되지 않음)
```

예시:

```
committedTail: "가", composing: "ㅅ"
입력: "획" (나랏글) → processor.input() 결과: committed "", composing "ㅈ"

→ hadPreviousComposing=true, committed="", composing.count==1
→ tryMerge종성("ㅈ", committedTail: "가")
→ automata.add글자("ㅈ", "가") → "갖" (1글자) → 성공!
→ 결과: committed "", composing "갖", consumedCommittedCount: 1
→ VC: delete("ㅅ") → delete("가") → insert("갖")
→ committedBuffer: "", composingBuffer: "갖"
```

### 3-2. deleteWithRestore종성

각 프로세서의 `delete()`를 호출한 뒤, 결과에 대해 종성 복원을 시도한다.

```
종성 복원 조건 (모두 만족해야 함):
1. 프로세서가 이미 committed를 소비하지 않았음 (consumedCommittedCount == 0)
2. result.composing이 낱자 자음 1개
3. committedTail이 비어있지 않음
4. isProtected가 false
```

예시:

```
committedTail: "가", composing: "나"
삭제 → processor.delete() → composing: "ㄴ" (모음 삭제)

→ consumedCommittedCount==0, composing.count==1
→ tryMerge종성("ㄴ", committedTail: "가")
→ "가" + "ㄴ" = "간" (1글자) → 성공!
→ 결과: composing "간", consumedCommittedCount: 1
→ VC: delete("나") → delete("가") → insert("간")
→ committedBuffer: "", composingBuffer: "간"
```

### 3-3. tryMerge종성 (공통 로직)

낱자 자음 1개를 committedTail의 마지막 한글에 종성으로 합치기를 시도하는 private 메서드.

```
1. 자음이 종성Table에 존재하고, 공백이 아닌지 확인
2. committedTail 마지막 글자가 분해 가능한 한글인지 확인
3. automata.add글자()로 결합 시도
4. 결과가 1글자이면 성공 (예: "가" + "ㄴ" → "간")
5. 2글자 이상이면 실패 → nil 반환
```

## 4. 컨트롤러의 핵심 메서드들

### 4-1. 글자 입력: `insertPrimaryKeyText`

```swift
open override func insertPrimaryKeyText(from button: TextInteractable) {
    guard let primaryKey = button.type.primaryKeyList.first else { ... }
    applyCompositionResult(
        processor.inputWithRestore종성(
            글자Input: primaryKey,
            composing: composingBuffer,
            committedTail: String(committedBuffer.suffix(2)),
            isProtected: committedBuffer.count <= protectedCommittedCount
        )
    )
}
```

프로세서의 `inputWithRestore종성`에 현재 composing, committedTail(최대 2글자), 보호 여부를 넘기고, 결과를 `applyCompositionResult`로 반영한다.

### 4-2. 결과 반영: `applyCompositionResult`

버퍼 관리의 핵심 메서드. 두 가지 경로가 있다.

#### 경로 A — 최적화 (committed == oldComposing && consumedCommittedCount == 0)

기존 composing이 그대로 확정된 경우. 이미 화면에 있으므로 지우고 다시 쓸 필요 없이 새 부분만 추가한다.

```
예: 천지인에서 is한글조합OnGoing=false일 때 새 자음 입력

composing: "가", 입력: "ㄴ"
→ committed: "가", composing: "ㄴ"
→ "가" == oldComposing "가" ✓ → proxy.insertText("ㄴ")만 실행
```

이 경로에서 `isPulledFromProtected`가 true이면 `protectedCommittedCount`를 복원하여 보호 상태를 유지한다.

#### 경로 B — 일반 경로

기존 composing을 전부 지우고, committed를 소비(종성 복원)한 뒤, 새 결과를 입력한다.

```
committedBuffer: "안녕", composing: "하"
입력: "ㄴ" → processor 결과: committed "", composing "한", consumedCommittedCount: 0

→ proxy: delete("하") → insert("한")
→ committedBuffer: "안녕", composingBuffer: "한"
```

종성 복원이 발생한 경우:

```
committedBuffer: "가", composing: "ㅅ"
입력: "획" → processor 결과: committed "", composing "갖", consumedCommittedCount: 1

→ proxy: delete("ㅅ") → applyConsumedCommitted(1) → delete("가")
→ proxy: insert("갖")
→ committedBuffer: "", composingBuffer: "갖"
```

`applyConsumedCommitted(count:)` 메서드가 `consumedCommittedCount`만큼 `committedBuffer`와 `textDocumentProxy`에서 글자를 제거한다.

### 4-3. 삭제: `deleteBackward`

세 가지 분기가 있다.

#### 분기 1 — composing이 있을 때

```
committedBuffer: "가", composingBuffer: "난"
삭제 → proxy: delete("난") → processor.deleteWithRestore종성("난", ...) → composing: "나"
→ proxy: insert("나") → composingBuffer: "나"
```

삭제 후 종성 복원이 일어나는 경우:

```
committedBuffer: "가", composingBuffer: "나"
삭제 → processor.deleteWithRestore종성("나", committedTail: "가", ...)
→ processor.delete() → "ㄴ"
→ 프로토콜 기본 구현에서 tryMerge종성("ㄴ", "가") → "간"
→ 결과: composing "간", consumedCommittedCount: 1
→ VC: delete("나") → applyConsumedCommitted(1) → delete("가") → insert("간")
→ committedBuffer: "", composingBuffer: "간"
```

composing이 완전히 비었으면 **`pullFromCommittedIfNeeded()`**를 호출하여 committed에서 끌어오기를 시도한다:

```
committedBuffer: "가나", composingBuffer: "ㄷ"
삭제 → processor 결과 → composing: "" (낱자 삭제)
→ pullFromCommittedIfNeeded()
→ committed 마지막 "나".isHangeul → true
→ committedBuffer: "가", composingBuffer: "나"
→ 보호 범위가 아니면 processor.start한글조합()
→ isPulledFromProtected = isProtected (보호 상태 기억)
```

#### 분기 2 — composing 비고 committed만 있을 때

committed의 마지막 글자를 꺼내서 삭제 처리한다.

```
committedBuffer: "가나", composingBuffer: ""
삭제 → "나".isHangeul → true
→ proxy: delete("나")
→ processor.deleteWithRestore종성("나", committedTail: "가", ...)
→ 결과: composing "ㄴ", consumedCommittedCount: 0 (또는 1이면 committed도 소비)
→ proxy: insert("ㄴ") → committedBuffer: "가", composingBuffer: "ㄴ"
→ 보호 범위가 아니면 processor.start한글조합()
```

비한글(숫자, 기호 등)이면 통째로 삭제한다.

#### 분기 3 — 둘 다 비었을 때

```
→ proxy.deleteBackward() (커서 앞의 시스템 텍스트 삭제)
```

### 4-4. 스페이스: `insertSpaceText`

```swift
let result = processor.inputSpace(composing: composingBuffer)
switch result {
case .insertSpace:       // 두벌식/나랏글: 실제 공백 입력 + 전체 버퍼 초기화
case .commitCombination: // 천지인: composing → committed 이동 + 보호 설정
}
```

`.commitCombination` 시 `protectedCommittedCount = committedBuffer.count`로 설정하여, 이후 삭제 시 보호된 글자에 종성이 합쳐지는 것을 방지한다.

### 4-5. 반복 입력 (길게 누르기): `repeatInsertPrimaryKeyText`

```swift
open override func repeatInsertPrimaryKeyText(from button: TextInteractable) {
    guard let lastInputText else { ... }
    if !composingBuffer.isEmpty { commitComposingBuffer() }
    textDocumentProxy.insertText(lastInputText)
    composingBuffer = lastInputText  // 마지막 글자를 composing에 유지
}
```

핵심: 마지막 글자를 composingBuffer에 남겨둠으로써, 반복 종료 후 다음 입력과 조합이 가능하다.

```
"ㄱ" 길게: ㄱ → ㄱㄱ → ㄱㄱㄱ (각각 committed로 이동, 마지막만 composing)
"ㅣ" 입력: composing "ㄱ" + "ㅣ" = "기" → 결과: "ㄱㄱ기"
```

### 4-6. 반복 삭제 + 종료

**반복 삭제 (`repeatDeleteBackward`)**: 글자 단위로 단순 삭제한다 (자소 분해 없음).

```swift
textDocumentProxy.deleteBackward()
composingBuffer.removeLast()  // 또는 committedBuffer.removeLast()
```

**반복 종료 (`repeatTextInteractionDidPerform`)**: DeleteButton일 때 composing이 비어있고 committed에 한글이 있으면 끌어오기를 실행한다.

```
committed: "가나", composing: "" (반복 삭제로 비워짐)
→ "나".isHangeul → committed에서 composing으로 이동
→ committedBuffer: "가", composingBuffer: "나"
→ 보호 범위가 아니면 processor.start한글조합()
```

### 4-7. protectedCommittedCount

천지인 전용 메커니즘. 천지인의 스페이스는 공백이 아닌 "조합 확정"이므로, 확정된 글자에 이후 삭제로 종성이 합쳐지면 안 된다.

```
"가" 입력 → 스페이스(확정) → "나" 입력
committedBuffer: "가나" 중 "가"까지가 보호 (protectedCommittedCount = 1)

삭제 → "나" 분해 → "ㄴ"
→ deleteWithRestore종성에서 tryMerge종성("ㄴ", committedTail: "가")
→ isProtected: committedBuffer.count(2) <= protected(1) → false → 시도
→ 그런데 VC에서 전달하는 isProtected는 committedBuffer.count <= protectedCommittedCount
→ 즉 2 <= 1 → false... 이 경우 실제로는 tryMerge가 시도됨

하지만 pullFromCommittedIfNeeded에서:
→ committedBuffer.count(1) <= protected(1) → true → isProtected
→ processor.start한글조합() 호출하지 않음 → 조합 재개 방지
```

끌어오기 시에도 보호 범위이면 `start한글조합()`을 호출하지 않아, 조합 재개를 방지한다.

### 4-8. isPulledFromProtected

`pullFromCommittedIfNeeded()`에서 보호된 글자를 끌어왔을 때 `true`로 설정되는 플래그.

이후 `applyCompositionResult`에서 해당 글자가 다시 committed로 돌아갈 때 `protectedCommittedCount`를 복원하여 보호 상태를 유지한다.

```
committedBuffer: "가" (보호됨, protectedCommittedCount=1), composingBuffer: ""
끌어오기 → committedBuffer: "", composingBuffer: "가", isPulledFromProtected = true

입력: "ㄴ" → processor 결과: committed "가", composing "ㄴ"
→ applyCompositionResult 경로 A (committed == oldComposing "가")
→ isPulledFromProtected가 true이므로 protectedCommittedCount 복원
→ committedBuffer: "가" (다시 보호됨), composingBuffer: "ㄴ"
```

## 5. 세 입력기의 차이점

### 5-1. 두벌식 (DubeolsikProcessor)

가장 단순한 입력기. 거의 모든 것을 오토마타에 위임한다.

```swift
func input(글자Input: String, composing: String) -> CompositionResult {
    let (committed, newComposing) = automata.add글자(글자Input: 글자Input, composing: composing)
    return CompositionResult(committed: committed, composing: newComposing, input글자: 글자Input)
}
```

|속성|값|
|---|---|
|`is한글조합OnGoing`|항상 `false` (스페이스가 항상 공백)|
|`start한글조합()` / `reset한글조합()`|빈 구현|
|`delete`|오토마타 삭제 후 composing 내부 종성 복원 시도|

#### delete 로직

두벌식의 `delete()`는 오토마타 삭제 후, composing이 2글자 이상이면 내부 종성 복원을 시도한다. `committedTail`/`isProtected`는 사용하지 않는다 (committed와의 종성 합치기는 프로토콜 기본 구현 `deleteWithRestore종성`에서 처리).

```
composing: "갈가" → automata.delete글자() → "갈ㄱ"
→ 마지막 "ㄱ"이 종성Table에 있음
→ automata.add글자("ㄱ", "갈") → "갉" (병합 성공)
→ 결과: composing "갉"
```

#### 입력 흐름 예시: "한글"

```
"ㅎ" → automata.add("ㅎ", "") → ("", "ㅎ")           composing: "ㅎ"
"ㅏ" → automata.add("ㅏ", "ㅎ") → ("", "하")          composing: "하"
"ㄴ" → automata.add("ㄴ", "하") → ("", "한")          composing: "한"
"ㄱ" → automata.add("ㄱ", "한") → ("한", "ㄱ")        committed→: "한", composing: "ㄱ"
"ㅡ" → automata.add("ㅡ", "ㄱ") → ("", "그")          composing: "그"
"ㄹ" → automata.add("ㄹ", "그") → ("", "글")          composing: "글"
```

### 5-2. 나랏글 (NaratgeulProcessor)

12키 기반. 기본 입력은 오토마타에 위임하되 **4가지 특수 기능**이 추가된다.

|속성|값|
|---|---|
|`is한글조합OnGoing`|항상 `false`|
|`start한글조합()` / `reset한글조합()`|빈 구현|
|`delete`|모음결합/이중모음 분해 시도 후 오토마타에 위임|

#### 특수 기능 1 — 획추가

composing의 마지막 글자에서 종성 → 중성 → 낱자 순으로 변환을 시도한다.

```
composing: "간", 입력: "획"
→ 종성 "ㄴ" → 획추가Table["ㄴ"] = "ㄷ"
→ "가" + "ㄷ" = "갇" → composing: "갇"
```

겹받침도 처리한다:

```
composing: "갉", 입력: "획"
→ 종성 "ㄺ" → decompose겹받침 → ("ㄹ", "ㄱ")
→ 뒷받침 "ㄱ" → 획추가 → "ㅋ"
→ "갈" + automata.add글자("ㅋ", "갈") → committed "갈", composing "ㅋ" (종성 불가 시)
```

#### 특수 기능 2 — 쌍자음

획추가와 동일한 구조로, 쌍자음Table을 사용한다.

```
composing: "간", 입력: "쌍"
→ 종성 "ㄴ" → 쌍자음Table["ㄴ"] = "ㄸ"
→ "ㄸ"은 종성Table에 없음 → "가" committed, "ㄸ" composing
```

#### 특수 기능 3 — 모음 토글 (ㅏ↔ㅓ, ㅗ↔ㅜ)

같은 모음 키를 반복하면 전환된다.

```
composing: "가", 입력: "ㅏ"
→ 중성 "ㅏ" → toggleTable["ㅏ"] = "ㅓ" → composing: "거"
→ 다시 "ㅏ" → "ㅓ" → toggleTable["ㅓ"] = "ㅏ" → composing: "가"
```

#### 특수 기능 4 — 'ㅣ' 결합 / 이중모음

```
composing: "가", 입력: "ㅣ"
→ 중성 "ㅏ" → 모음결합Table["ㅏ"] = "ㅐ" → composing: "개"

composing: "무", 입력: "ㅓ"
→ 중성 "ㅜ" → 이중모음결합Table["ㅜ"] = "ㅝ" → composing: "뭐"
```

#### 입력 처리 우선순위

```
1. "획" / "쌍" → 특수 변환
2. "ㅣ" → 모음 결합 시도 (실패하면 일반 입력으로)
3. "ㅏ"/"ㅓ" → 이중모음 시도 (실패하면 4로)
4. ㅏ/ㅓ/ㅗ/ㅜ → 토글 시도 (실패하면 5로)
5. 일반 입력 → automata.add글자()에 위임
```

#### 삭제도 역순으로 특수 처리

```
1. 'ㅣ' 결합 분해 (ㅐ → ㅏ)
2. 이중모음 분해 (ㅝ → ㅜ)
3. 일반 삭제 → automata.delete글자()
```

나랏글의 `delete()`는 `committedTail`/`isProtected`를 사용하지 않는다. committed와의 종성 합치기는 프로토콜 기본 구현에서 처리한다.

### 5-3. 천지인 (CheonjiinProcessor)

가장 복잡한 입력기. 세 가지 고유 메커니즘이 있다.

|속성|값|
|---|---|
|`is한글조합OnGoing`|동적 관리 (`true`/`false` 전환)|
|`start한글조합()`|`is한글조합OnGoing = true`|
|`reset한글조합()`|`is한글조합OnGoing = false`|
|`delete`|비표준 모음(ㆍ, ᆢ) 처리 + committed 끌어오기|

#### 고유 메커니즘 1 — `is한글조합OnGoing` 상태 관리

두벌식/나랏글과 달리 천지인은 이 플래그를 적극적으로 사용한다.

```
false 상태에서 한글 입력 → 기존 composing 전체 확정 + 새 글자로 조합 시작 + true로 전환
true 상태에서 한글 입력  → 기존 composing과 조합 계속
스페이스 입력            → commitCombination + false로 전환
```

이것이 필요한 이유: 천지인에서 자음 순환 중 "다른 글자"와 조합해야 하는지, "새 글자"인지 구분해야 하기 때문. 스페이스로 조합을 끊으면 다음 입력은 새 글자.

#### 고유 메커니즘 2 — 자음 순환

같은 자음 키를 반복하면 순환한다.

```
"ㄱ" 순환: ㄱ → ㅋ → ㄲ → ㄱ → ...
"ㄴ" 순환: ㄴ → ㄹ → ㄴ → ...
"ㅇ" 순환: ㅇ → ㅁ → ㅇ → ...
```

종성에서도 순환한다:

```
composing: "각", 입력: "ㄱ"
→ 종성 "ㄱ" → cycleList에 있음 → 다음: "ㅋ"
→ "가" + "ㅋ" = "갘"? → 종성Table에 "ㅋ" 있으므로 composing: "갘"
```

겹받침의 뒷받침도 순환 대상이다:

```
composing: "갃", 입력: "ㅅ"  (종성 "ㄳ")
→ decompose겹받침("ㄳ") → ("ㄱ", "ㅅ")
→ "ㅅ"이 cycleList["ㅅ"]에 있음 → 다음: "ㅎ"
→ "각" + automata.add글자("ㅎ", "각") → composing: 결합 결과
```

#### 고유 메커니즘 3 — 천지인 모음 조합 (ㆍ/ㅡ/ㅣ)

3개의 키(천ㆍ/지ㅡ/인ㅣ)를 조합하여 21개 모음을 만든다.

```
모음조합Table 예시:
"ㅣ" + "ㆍ" = "ㅏ"
"ㅏ" + "ㆍ" = "ㅑ"
"ㆍ" + "ㆍ" = "ᆢ"
"ᆢ" + "ㅣ" = "ㅕ"
"ㅡ" + "ㆍ" = "ㅜ"
"ㆍ" + "ㅡ" = "ㅗ"
```

비표준 문자(ㆍ, ᆢ)는 중간 상태로만 존재하며, 표준 모음이 완성되면 오토마타에 위임된다.

```
입력: "ㅣ" → composing: "ㅣ" (표준 모음, automata에 위임)
입력: "ㆍ" → tryCombine모음("ㅣ", "ㆍ") → "ㅏ" (표준 모음)
          → automata.add글자("ㅏ", "") → composing: "ㅏ"
입력: "ㆍ" → tryCombine모음("ㅏ", "ㆍ") → "ㅑ" (표준 모음)
          → automata.add글자("ㅑ", "") → composing: "ㅑ"
```

완성형 글자 안의 중성도 조합한다:

```
composing: "가", 입력: "ㆍ"
→ tryCombine모음: decompose("가") → 초성 "ㄱ", 중성 "ㅏ", 종성 없음
→ 모음조합Table["ㅏ"]["ㆍ"] = "ㅑ" (표준 모음)
→ prefix: "ㄱ" (초성만 분리)
→ automata.add글자("ㅑ", "ㄱ") → composing: "갸"
```

#### 천지인 delete() 고유 로직

천지인의 `delete()`는 두 가지 고유 처리를 수행한다:

**1. 비표준 모음 직접 처리**: composing에 ㆍ이나 ᆢ가 포함되어 있으면 오토마타에 위임하지 않고 마지막 글자만 단순 제거한다 (오토마타는 비표준 모음을 처리할 수 없으므로).

```
composing: "간ㆍ" → 마지막 "ㆍ" 제거 → composing: "간"
```

**2. committed 비표준 모음 끌어오기**: 삭제 후 남은 composing이 자음 0~1개이고, committedTail 끝이 비표준 모음(ㆍ, ᆢ)이면 committedTail 전체를 composing으로 올린다.

```
committedTail: "간ㆍ", composing: "ㄷ"
삭제 후 → composing: "" → tryRestore비표준모음ToCommitted
→ committedTail 끝이 빈 경우는 처리하지 않지만...

실제로는:
committedTail: "간ㆍ", composing이 자음 1개 "ㄷ" (삭제 전 상태가 아닌 오토마타 삭제 결과)
→ committedTail 마지막 "ㆍ"가 비표준모음Set에 있음
→ fullComposing = "간ㆍ" + "ㄷ" = "간ㆍㄷ"
→ 결과: composing "간ㆍㄷ", consumedCommittedCount: 2
→ 이후 모음 입력(예: ㆍ+ㅣ=ㅓ)으로 "간" → "가너" 연음 가능
```

표준 한글과의 종성 합치기는 프로토콜 기본 구현 `deleteWithRestore종성`에서 처리한다.

#### 천지인 input() 전체 흐름

```
1. 비한글 → 확정 + is한글조합OnGoing = false
2. is한글조합OnGoing == false → composing 확정 + 새 글자 시작 + true
3. 모음 입력 (ㆍ/ㅡ/ㅣ 또는 표준 모음):
   A. tryCombine모음 시도 → 성공하면 표준/비표준 분기
   B. 실패하면 단순 추가 (표준이면 automata, 비표준이면 직접 append)
4. 자음 입력:
   → 순환Table에 있으면 cycle자음()
   → 없으면 automata.add글자()
```

## 6. 오토마타 (HangeulAutomata)

프로세서들이 공통으로 사용하는 **표준 한글 자소 합치기/분해 엔진**.

### add글자 — 글자 합치기

```
입력이 모음인 경우:
  1-1. 종성 있는 완성형 + 모음 → 연음 ("각" + "ㅏ" → "가" + "가")
       겹받침이면 분리 후 연음 ("닭" + "ㅏ" → "달" + "가")
  1-2. 종성 없는 완성형 + 모음 → 겹모음 시도 ("고" + "ㅏ" → "과")
  낱자 자음 + 모음 → 결합 ("ㄱ" + "ㅏ" → "가")
  낱자 모음 + 모음 → 겹모음 시도 ("ㅜ" + "ㅓ" → "ㅝ")

입력이 자음인 경우:
  2-1. 종성 없는 완성형 + 자음 → 종성 추가 ("가" + "ㄱ" → "각")
  2-2. 종성 있는 완성형 + 자음 → 겹받침 시도 ("각" + "ㅅ" → "갃")

위 모든 조건 실패 → composing 전체 확정 + 새 글자 시작
```

### delete글자 — 글자 분해

```
완성형 한글:
  종성 있음 → 겹받침이면 앞부분만 ("닭" → "달"), 홑받침이면 제거 ("각" → "가")
  종성 없음 → 겹모음이면 앞부분만 ("과" → "고"), 단모음이면 초성만 ("가" → "ㄱ")

낱자:
  겹모음이면 분해 ("ㅘ" → "ㅗ"), 아니면 그냥 삭제
```

### combine / decompose — 유니코드 연산

```
한글 유니코드 = 0xAC00 + (초성 × 21 × 28) + (중성 × 28) + 종성
```

이 공식을 이용해 초성/중성/종성 인덱스 ↔ 완성형 글자를 변환한다.
