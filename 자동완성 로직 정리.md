## 전체 아키텍처

```
사용자 터치
    ↓
BaseKeyboardViewController (버튼 이벤트 라우팅)
    ↓
SuggestionController (SuggestionService 프로토콜 구현)
    ├── LexiconPredictiveTextEngine     ← UILexicon 기반 (텍스트 대치)
    ├── TextCheckerPredictiveTextEngine  ← UITextChecker 기반 (시스템 사전 + 오타 교정)
    └── NGramPredictiveTextEngine        ← 사용자 이력 기반 (다음 단어 예측)
    ↓
SuggestionBarHStackView (3개 버튼 표시)
    ├── button1  ← "현재입력" 또는 n-gram 예측 1
    ├── button2  ← 자동완성 후보 또는 n-gram 예측 2
    └── button3  ← 자동완성 후보 또는 n-gram 예측 3
```

핵심 원칙:
- **두 가지 모드가 완전히 분리**되어 동작한다: 입력 중 모드(lexicon + textChecker)와 n-gram 모드(bigram + trigram).
- **엔진은 후보 생성만 담당하고, 텍스트 조작(삭제/삽입)은 VC가 전담**한다.
- **학습 정책은 출처별로 다르다**: textChecker 후보만 `UITextChecker.learnWord`로 학습하고, lexicon 후보는 시스템이 관리하므로 학습하지 않는다.

## 1. 핵심 엔진 구조

세 엔진 모두 `PredictiveTextProvider` 프로토콜을 채택한다.

```swift
protocol PredictiveTextProvider: AnyObject {
    func suggestions(for contextBeforeInput: String) -> [String]
    func learn(word: String)
}
```

### 1-1. LexiconPredictiveTextEngine

`UILexicon` 기반. 사용자의 연락처, 텍스트 대치 항목 등 개인화된 데이터를 사용한다.

|항목|내용|
|---|---|
|데이터 소스|`UILexicon` (시스템 관리)|
|매칭 방식|현재 입력 단어와 `entry.userInput`이 **정확히 일치**할 때만 반환|
|학습|불필요 (시스템이 관리)|
|로딩 시점|`viewDidLoad`에서 `requestSupplementaryLexicon` 호출|

```
입력: "ㅈㄱㅈ" → lexicon에 userInput="ㅈㄱㅈ", documentText="지금 가는 중!" 존재
→ 정확히 일치 → ["지금 가는 중!"] 반환
```

iOS 버그 대응으로 `'m' → 'M'` 매핑은 제외한다.

### 1-2. TextCheckerPredictiveTextEngine

`UITextChecker` 기반. 시스템 사전을 사용하며 한국어와 영어를 동시에 지원한다.

|항목|내용|
|---|---|
|데이터 소스|시스템 사전 + `UITextChecker.learnWord`로 학습된 단어|
|매칭 방식|1순위 `completions` (접두어 자동완성), 2순위 `guesses` (오타 교정)|
|학습|`UITextChecker.learnWord` (사전에 단어 등록)|
|중복 제거|두 소스의 결과를 합칠 때 lowercased 기준으로 중복 제거|

```
completions: "hel" → ["hello", "help", "helmet", ...]  (접두어 매칭)
guesses:     "helo" → ["hello", "hero", ...]            (오타 교정)
```

두 소스는 항상 순차적으로 합쳐진다. completions 결과가 있어도 guesses로 보충하여 다양한 후보를 제공한다.

### 1-3. NGramPredictiveTextEngine

사용자 입력 이력 기반. bigram(직전 1단어)과 trigram(직전 2단어)의 빈도를 기록하여 다음 단어를 예측한다.

|항목|내용|
|---|---|
|데이터 소스|사용자 입력 이력 (직접 기록)|
|저장소|App Group `UserDefaults`|
|매칭 방식|1순위 trigram, 2순위 bigram, 빈도순 정렬|
|학습|`addWord(_:)`로 시퀀스 기록|
|제한|키당 최대 50개 항목, 전체 최대 5000개 키|

```
bigramStore = { "오늘": {"날씨": 2, "점심": 1} }
trigramStore = { "오늘 날씨": {"좋다": 2} }

입력: "오늘 날씨 " → trigram["오늘 날씨"] = ["좋다"], bigram["날씨"] = ["어때"]
→ 결과: ["좋다", "어때"]
```

## 2. SuggestionController — 통합 관리

`SuggestionService` 프로토콜을 구현하며, 세 엔진의 결과를 병합하고 텍스트 대치/복구를 관리한다.

### 2-1. 두 가지 표시 모드

`performUpdateSuggestions`에서 문맥에 따라 모드를 분기한다.

```
let context = contextBeforeInput ?? ""

if context.isEmpty || context.last?.isWhitespace == true {
    → n-gram 모드: button1~3에 다음 단어 예측
} else {
    → 입력 중 모드: button1에 "현재단어", button2~3에 자동완성 후보
}
```

|조건|모드|사용 엔진|버튼 배치|
|---|---|---|---|
|입력 없음 / 마지막이 공백|n-gram 모드|trigram → bigram|button1~3: 예측 후보|
|단어 타이핑 중|입력 중 모드|lexicon → textChecker|button1: "현재단어", button2~3: 후보|

두 모드는 완전히 분리되어 있고, 동시에 섞이지 않는다.

### 2-2. 입력 중 모드 — 후보 병합

`mergeSuggestions(for:currentWord:)`에서 lexicon과 textChecker 결과를 병합한다.

```
1. seen에 currentWord를 먼저 등록 (button1과의 중복 방지)
2. lexicon 결과를 먼저 추가 (텍스트 대치 우선)
3. 남은 슬롯을 textChecker 결과로 채움
4. 최대 2개 반환 (button2~3용)
```

예시:

```
currentWord: "ㅈㄱㅈ"
seen: {"ㅈㄱㅈ"}

lexicon 결과: ["지금 가는 중!"] → seen에 없음 → 추가
textChecker 결과: ["적극적"] → seen에 없음 → 추가

최종: [SuggestionItem("지금 가는 중!", .lexicon), SuggestionItem("적극적", .textChecker)]
button1: "ㅈㄱㅈ", button2: "지금 가는 중!", button3: "적극적"
```

currentWord와 동일한 후보는 **생성 단계에서 제외**되므로, 나중에 빈 슬롯이 발생하지 않는다.

### 2-3. n-gram 모드 — 다음 단어 예측

`nGramSuggestions(for:)`에서 n-gram 엔진의 결과를 그대로 사용한다.

```
1. trigram 조회 (직전 2단어 문맥)
2. bigram 조회 (직전 1단어 문맥)로 부족한 슬롯 보충
3. 중복 제거, 빈도순 정렬
4. 최대 3개 반환 (button1~3 모두 사용)
```

사용 이력이 없으면 결과가 비어있고, 세 버튼 모두 빈 상태로 표시된다.

### 2-4. 후보 선택 — selectSuggestion

사용자가 button2~3을 탭했을 때 호출된다.

```swift
func selectSuggestion(at index: Int, ...) -> (deleteCount: Int, insertText: String)?
```

|동작|설명|
|---|---|
|deleteCount 계산|현재 입력 중인 단어의 글자 수|
|insertText 결정|선택된 후보의 텍스트|
|학습|textChecker 후보만 `UITextChecker.learnWord`|
|대치 이력|lexicon 후보일 때만 `replacementHistory`에 기록|

```
context: "hel", 선택: "hello"
→ deleteCount: 3 ("hel" 삭제), insertText: "hello"
→ VC: proxy.deleteBackward() × 3 → proxy.insertText("hello")
```

### 2-5. 텍스트 대치 — attemptTextReplacement

스페이스 입력 시 자동으로 호출된다. `UILexicon`의 `userInput`과 정확히 매칭되면 자동 대치를 수행한다.

```
context: "ㅈㄱㅈ"
lexicon에 userInput="ㅈㄱㅈ", documentText="지금 가는 중!" 존재
→ 가장 긴 매칭 우선
→ ignoredShortcut와 동일하면 건너뜀 (재대치 방지)
→ 대치 이력(replacementHistory)에 기록
→ (deleteCount: 3, insertText: "지금 가는 중!")
```

`selectSuggestion`에서 button2의 lexicon 후보를 직접 탭한 것과 동일한 효과이지만, 이쪽은 스페이스 입력 시 **자동**으로 동작한다.

### 2-6. 대치 복구 — attemptRestoreReplacement

삭제 입력 시 호출된다. 방금 대치된 텍스트를 원래 단축어로 되돌린다.

```
context: "지금 가는 중!"
replacementHistory에 (userInput: "ㅈㄱㅈ", documentText: "지금 가는 중!") 존재
→ context.hasSuffix("지금 가는 중!") → 일치
→ (deleteCount: 7, insertText: "ㅈㄱㅈ")
→ ignoredShortcut = "ㅈㄱㅈ" (재대치 방지)
```

복구 후 다시 스페이스를 누르면, `ignoredShortcut`에 의해 같은 단축어의 재대치가 방지된다.

## 3. SuggestionBar 표시 로직

`SuggestionBarHStackView`는 3개의 `WordSuggestionButton`과 2개의 디바이더로 구성된다.

### 3-1. 입력 중 모드의 버튼 배치

`currentWord`가 존재할 때:

|버튼|표시 내용|예시|
|---|---|---|
|button1|`"현재단어"` (따옴표로 감싸기)|`"hel"`|
|button2|lexicon 후보 우선, 없으면 textChecker|`hello`|
|button3|남은 후보|`help`|

### 3-2. n-gram 모드의 버튼 배치

`currentWord`가 nil일 때:

|버튼|표시 내용|예시|
|---|---|---|
|button1|n-gram 예측 1순위 (trigram 우선)|`좋다`|
|button2|n-gram 예측 2순위|`어때`|
|button3|n-gram 예측 3순위 (부족하면 비움)|(비어있음)|

### 3-3. 버튼 탭 처리

`SuggestionBarDelegate`를 통해 `BaseKeyboardViewController`에서 처리한다.

**n-gram 모드에서 탭:**

```
button1 탭 (index 0) → 해당 예측 단어 삽입 + 스페이스 + n-gram 기록
button2 탭 (index 1) → 동일
button3 탭 (index 2) → 동일
→ 이후 다시 n-gram 모드로 갱신
```

**입력 중 모드에서 탭:**

```
button1 탭 (index 0) → 현재 단어 확정 + learnWord + recordWord + 후보 비움
button2 탭 (index 1) → selectSuggestion(0) → 현재 단어를 후보로 교체
button3 탭 (index 2) → selectSuggestion(1) → 현재 단어를 후보로 교체
```

button2~3 탭 시 인덱스가 `index - 1`로 변환되어 `selectSuggestion`에 전달된다.

## 4. 학습 정책

### 4-1. UITextChecker 학습

`UITextChecker.learnWord`는 단어를 시스템 사전에 등록한다. 빈도 카운팅이나 우선순위 관리는 없으며, 한 번 학습하면 이후 completions/guesses 후보에 포함된다.

|동작|학습 여부|이유|
|---|---|---|
|button1 탭 (현재 입력 확정)|✅|사용자가 의도한 단어|
|button2~3 textChecker 후보 선택|✅|사용자 선호 반영|
|button2~3 lexicon 후보 선택|❌|UILexicon이 이미 관리, 중복 학습 방지|
|n-gram 예측 후보 선택|❌|n-gram 엔진이 별도 관리|

### 4-2. n-gram 기록

`addWord(_:)`로 단어 시퀀스를 기록한다. 스페이스 입력 시 직전 단어, button1 탭 시 확정 단어, button2~3 탭 시 선택 후보가 기록 대상이다.

|동작|기록 여부|설명|
|---|---|---|
|스페이스 입력|✅|직전 단어를 기록|
|button1 탭 (현재 입력 확정)|✅|확정 단어를 기록|
|button2~3 후보 선택|✅|선택한 후보를 기록|
|리턴 입력|문장 종료|`endSentence()`로 버퍼 초기화|

## 5. n-gram 저장 구조

### 5-1. 저장 과정

스페이스를 누를 때마다 `addWord`가 호출되어 단어가 문장 버퍼에 쌓이고, 동시에 n-gram이 기록된다.

```
"오늘 날씨 좋다" 입력 과정:

[스페이스 1] addWord("오늘")
  버퍼: ["오늘"]
  → 단어 1개 → bigram/trigram 기록 없음

[스페이스 2] addWord("날씨")
  버퍼: ["오늘", "날씨"]
  → bigram: "오늘" → {"날씨": 1}

[스페이스 3] addWord("좋다")
  버퍼: ["오늘", "날씨", "좋다"]
  → bigram: "날씨" → {"좋다": 1}
  → trigram: "오늘 날씨" → {"좋다": 1}
```

같은 패턴을 반복하면 빈도가 증가한다:

```
두 번째 "오늘 날씨 좋다" 입력 후:

bigramStore = {
    "오늘": {"날씨": 2},        ← 빈도 1→2
    "날씨": {"좋다": 2},        ← 빈도 1→2
}
trigramStore = {
    "오늘 날씨": {"좋다": 2}    ← 빈도 1→2
}
```

### 5-2. 조회 과정

`suggestions(for:)`에서 문맥의 마지막 단어들을 키로 사용한다.

```
context: "오늘 날씨 "

[1단계] trigram 조회: key = "오늘 날씨"
  → trigramStore["오늘 날씨"] = {"좋다": 2}
  → 결과: ["좋다"]

[2단계] bigram 조회: key = "날씨"
  → bigramStore["날씨"] = {"좋다": 2, "어때": 1}
  → "좋다"는 이미 trigram에서 나왔으므로 중복 제거
  → 추가: ["어때"]

[최종] ["좋다", "어때"]
```

trigram이 더 긴 문맥을 반영하므로 우선순위가 높고, bigram은 부족한 슬롯을 보충한다.

### 5-3. 항목 수 제한

데이터가 무한히 커지는 것을 방지하기 위해 두 가지 제한을 둔다.

|제한|값|동작|
|---|---|---|
|키당 최대 항목 수|50개|초과 시 빈도 낮은 항목부터 제거|
|전체 최대 키 수|5000개|초과 시 총 빈도가 낮은 키부터 제거|

정리(`pruning`)는 n-gram 기록 시마다 수행된다.

### 5-4. 디스크 저장 시점

매번 기록할 때마다 디스크에 쓰면 성능이 떨어지므로 디바운스를 적용한다.

|시점|동작|
|---|---|
|10회 기록마다|자동 저장 (`scheduleSave`)|
|키보드 비활성화 시|`viewWillDisappear`에서 `saveNGramData` 호출|
|수동 호출|`resetNGramData`로 전체 초기화 가능|

저장소는 App Group `UserDefaults`를 사용하여 메인 앱과 키보드 익스텐션 모두에서 접근 가능하다.

## 6. BaseKeyboardViewController — 호출 시점

### 6-1. 키 입력

```swift
// textInteractionWillPerform
if !(button is SpaceButton) {
    suggestionController.clearIgnoredShortcut()  // 재대치 방지 상태 초기화
}

// textInteractionDidPerform
if !isRepeatingInput && isPredictiveTextEnabled {
    suggestionController.updateSuggestions(contextBeforeInput: ...)
}
```

모든 키 입력 후 후보가 갱신된다. 스페이스 이외의 키를 누르면 `ignoredShortcut`이 초기화되어 다음 스페이스에서 텍스트 대치가 다시 가능해진다.

### 6-2. 스페이스 입력

```swift
// performTextInteraction — .spaceButton 분기
if isTextReplacementEnabled,
   let replacement = suggestionController.attemptTextReplacement(contextBeforeInput: ...) {
    // 텍스트 대치 수행: deleteCount만큼 삭제 → insertText 삽입
}

// insertSpaceText
let lastWord = context에서 마지막 단어 추출
suggestionController.recordWord(lastWord)  // n-gram 기록
textDocumentProxy.insertText(" ")
```

스페이스 입력 시 두 가지가 순차적으로 발생한다: 텍스트 대치 시도(선행), n-gram 기록 + 공백 삽입(후행).

### 6-3. 삭제 입력

```swift
// performTextInteraction — .deleteButton 분기
if isTextReplacementEnabled,
   let restore = suggestionController.attemptRestoreReplacement(contextBeforeInput: ...) {
    // 대치 복구: deleteCount만큼 삭제 → insertText(원래 단축어) 삽입
} else {
    deleteBackward()  // 일반 삭제
}
```

삭제 시 먼저 대치 복구를 시도하고, 복구할 대상이 없으면 일반 삭제를 수행한다.

### 6-4. 리턴 입력

```swift
// insertReturnText
textDocumentProxy.insertText("\n")
suggestionController.clearReplacementHistory()  // 대치 이력 초기화
suggestionController.endSentence()              // n-gram 문장 버퍼 초기화
```

리턴 입력은 문장의 끝을 의미하므로, 대치 이력과 n-gram 문장 버퍼를 모두 초기화한다.

### 6-5. SuggestionBar 표시/숨김

`updateSuggestionBarHidden()`에서 제어한다.

```
숨김 조건 (하나라도 만족하면 숨김):
1. isPredictiveTextEnabled == false
2. textDocumentProxy.autocorrectionType == .no
3. currentKeyboard == .tenKey
```

숨겨지면 `suggestionController.isEnabled = false`로 설정되어 연산 자체를 건너뛴다. 표시 상태가 변경되면 키보드 높이도 재계산한다.

## 7. 상태 관리

### 7-1. ignoredShortcut — 재대치 방지

대치 복구 후 같은 단축어가 다시 대치되는 것을 방지한다.

```
"ㅈㄱㅈ" 입력 → 스페이스 → "지금 가는 중!"으로 자동 대치
→ 삭제 → 복구: "지금 가는 중!" → "ㅈㄱㅈ"
→ ignoredShortcut = "ㅈㄱㅈ"

→ 다시 스페이스 → attemptTextReplacement에서 ignoredShortcut 확인 → 대치 건너뜀
→ ignoredShortcut = nil로 초기화

→ 스페이스 이외의 키 입력 → clearIgnoredShortcut() → nil로 초기화
```

### 7-2. replacementHistory — 대치 이력

텍스트 대치가 수행될 때마다 `(userInput, documentText)` 쌍을 기록한다. 삭제 시 `context.hasSuffix(documentText)`로 역순 검색하여 가장 최근 대치를 복구한다.

```swift
struct ReplacementRecord: Equatable {
    let userInput: String      // "ㅈㄱㅈ"
    let documentText: String   // "지금 가는 중!"
}
```

기록되는 시점:
- `attemptTextReplacement` — 스페이스 자동 대치 시
- `selectSuggestion` — lexicon 후보 직접 선택 시

초기화되는 시점:
- `insertReturnText` — 리턴 입력 시 (`clearReplacementHistory`)

### 7-3. isEnabled — 활성화 제어

`false`로 설정하면 모든 후보 연산을 건너뛰고 현재 후보를 즉시 초기화한다.

```swift
var isEnabled: Bool = true {
    didSet {
        if !isEnabled { clearSuggestions() }
    }
}
```

`updateSuggestionBarHidden()`에서 SuggestionBar가 숨겨질 때 `false`로 설정된다.

## 8. 메인 앱 — 학습 데이터 초기화

`PredictiveTextSettingsView`에서 자동완성 활성화 토글과 학습 데이터 초기화를 제공한다.

### 초기화 경로

메인 앱에서는 `SuggestionController`를 거치지 않고 엔진을 직접 생성하여 초기화한다.
`SuggestionService` 프로토콜에 초기화 메서드를 두지 않는 이유는,
초기화는 설정 앱 전용 동작이고 키보드 런타임과 무관하기 때문이다.
```swift
// TextChecker 학습 단어 초기화
let textCheckerEngine = TextCheckerPredictiveTextEngine(languages: ["ko_KR", "en-US"])
textCheckerEngine.unlearnAllWords()

// n-gram 예측 데이터 초기화
let nGramEngine = NGramPredictiveTextEngine()
nGramEngine.resetAllData()
```

두 엔진 모두 App Group `UserDefaults`를 사용하므로,
메인 앱에서 초기화하면 다음 키보드 세션부터 반영된다.

### TextChecker 학습 이력 관리

`UITextChecker.learnWord`로 학습한 단어는 `UITextChecker.unlearnWord`로만 개별 삭제 가능하다.
일괄 초기화를 위해 학습한 단어 목록을 App Group `UserDefaults`에 별도 저장한다.

|시점|동작|
|---|---|
|`learn(word:)`|`UITextChecker.learnWord` + `learnedWords`에 추가|
|`unlearnAllWords()`|`learnedWords` 순회하며 `UITextChecker.unlearnWord` + 저장소 초기화|

중복 학습 방지를 위해 `UITextChecker.hasLearnedWord`로 사전 체크한다.
